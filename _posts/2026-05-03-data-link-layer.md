> 物理层交给你一串连续的比特流——没有边界，没有检错，没有"谁该发、谁该收"。链路层就是来解决这些问题的。

---

## 逻辑链总览

```
物理层交给你：一串连续的比特流（可能某些比特被噪声翻转过）
  ↓
问题 1：这一串比特里，从哪开始算"一个完整的数据块"？→ 帧定界
  ↓
问题 2：这个数据块中间有没有比特被翻转过？→ 差错检测（CRC）
  ↓
问题 3：出错了怎么办？→ 重传（ARQ 协议）
  ↓
问题 4：多个设备共享同一根线，谁先发？→ 介质访问控制（CSMA/CD, CSMA/CA）
  ↓
问题 5：局域网里有 N 台设备，发给谁？→ MAC 地址 + 交换机自学习 + ARP
```

---

## 问题 1：帧定界——从连续比特流里切出"帧"

物理层给你的是这样：`...01001011100101010010010100100101001010010...`——连续、无边界、无限长。

你不知道前面的 `01001` 是一个独立的帧，还是上一帧的尾巴。你必须在比特流里自己标出"这里开始、这里结束"。

### 方案一：字节计数法（最直接，但致命缺陷）

帧头里写"数据长度 = 100"，接收方读 100 字节后停下。**致命问题**：长度字段本身可能被噪声翻转过。长度从 100 翻成 200 → 读了 200 字节才停 → 后续所有帧全部错位。一个比特错，全链崩溃。

### 方案二：字符填充法（PPP 协议的做法）

用 `0x7E` 做帧边界标记。数据里碰巧有 `0x7E` 怎么办？→ 转义：`0x7E` 替换成 `0x7D 0x5E`，就像编程里 `\n` 写成 `\\n`。

### 方案三：比特填充法（链路层最经典的方案）

```
用 01111110（连续 6 个 1，两端各一个 0）作为帧边界标志

发送方边发边数：每看到数据里有连续 5 个 1 → 自动在后面塞一个 0
  数据：     ...111110...
  实际发送： ...1111100... ← 塞的 0

接收方边收边数：每看到连续 5 个 1 后面跟一个 0 → 删掉那个 0
  看到 01111110 → 这是帧边界，不是数据
```

关键：填充和去填充对上层完全透明。上层交什么数据，对方上层就收到什么数据。

## 问题 2：差错检测——怎么知道帧被传错了

帧定界告诉你"一帧在哪"。但里面的比特对不对？物理层的热噪声总会翻转一些比特。

### 奇偶校验（最简单但最弱）

在数据后加一个比特，让整个串里 1 的个数是奇数或偶数。只能检出**奇数个**错误——两个比特同时翻转，互相抵消，校验通过。漏检率太高。

### CRC——所有链路层协议的实际选择

**核心思想**：把数据当成一个巨大的二进制数，除以一个固定的"生成多项式"，余数就是 CRC 校验值。发方附在数据后，收方重新除一遍——余数不匹配，肯定有错。

```
发送方：数据 D → 在末尾加 k 个 0 → 除以生成多项式 G → 得余数 R
       发送：[D][R]

接收方：收到 [D'][R'] → 除以同一个 G
       余数为 0 → 无误
       余数不为 0 → 有错，丢弃
```

CRC 能检出的错误：所有单个比特错误、所有奇数个比特错误、所有长度 ≤ k 的突发错误、绝大多数 > k 的突发错误。

CRC-32（以太网 FCS 字段）：每 2³² 个随机错误模式中，只有 1 个能骗过 CRC。

## 问题 3：出错了怎么办？——ARQ 协议

检错是"知道错了"。然后呢？答案是让发送方重传。

### 停等协议（Stop-and-Wait）

发一帧 → 等 ACK → 收到 ACK 再发下一帧。超时未收到 ACK → 重发。

**致命缺陷**：发一帧等一帧。1Mbps 链路，RTT=20ms，每帧 1000 字节：

```
发送一帧 = 8ms，等 ACK = 20ms
信道利用率 = 8 / (8+20) = 28%
链路 72% 的时间在空转！
```

### 后退 N 帧（Go-Back-N）

**洞察**：不用等一个帧的 ACK 回来再发下一个。维护一个"发送窗口"，连续发 N 个帧。

出错了：帧 2 丢了 → 接收方丢弃帧 3、4（它们是乱序的）→ 发送方收到 3 个重复 ACK 1 → 重发帧 2, 3, 4（全部重发）。一个帧出错，后面的正确帧也被牵连。

### 选择重传（Selective Repeat）

**洞察**：帧 2 丢了，为什么帧 3、4 也要重发？

接收方为每个帧单独缓存。帧 2 丢了，帧 3、4 先缓存住。发送方只重发帧 2。代价：接收方需要缓冲区来存乱序帧。

| | 停等 | 后退 N 帧 | 选择重传 |
|---|---|---|---|
| 信道利用率 | 最低 | 高 | 最高 |
| 缓冲区需求 | 最低 | 低 | 高 |
| 实际应用 | 早期简单协议 | 偶尔 | **TCP 的选择确认 (SACK)** |

TCP 混合用了 GBN 和 SR——基本确认是累积确认（GBN 风格），加上 SACK 选项后做选择性重传（SR 风格）。

## 问题 4：多个设备共享同一根线——谁先发？

前面的讨论假设 A 和 B 独占一根线。但在早期以太网和 Wi-Fi 里，多台设备共享同一信道。同时发 → 信号碰撞 → 全毁。

### CSMA/CD（有线以太网）——"先听再说，边说边听"

1. **载波监听（先听）**：发之前先听 → 有人在发？等。没人？发。
2. **冲突检测（边说边听）**：发的同时继续听 → 听到信号跟自己发的不一样？撞了！立刻停发。
3. **二进制指数退避**：第 k 次撞 → 从 `{0,...,2^k-1}` 里随机选等待时间。越撞得多，等待范围越大。

**现代以太网为什么不再有冲突？** 交换机替代了集线器。交换机的每个端口是独立的冲突域，信号在交换机内部被缓存转发，不在同一根线上，永远不会互相干扰。

### CSMA/CA（Wi-Fi）——"先听再说，但没法边发边听"

Wi-Fi 不能做冲突检测，因为自己的发射功率远超远方信号，**你听不到别人的信号**。所以用"碰撞避免"：

1. **DIFS**：信道空闲后等一段时间
2. **随机退避**：从 0~CW 中随机挑一个数 → 倒计时到 0 → 发
3. **ACK**：没收到 ACK → 认为碰撞了 → 增大 CW → 重新退避

Wi-Fi 的"不稳定"很大一部分来自这个退避机制——拥挤环境中（30 台设备连同一个路由器），CW 被撑得很大，每个人的平均等待时间暴涨。

## 问题 5：局域网里有 N 台设备，发给谁？

### MAC 地址

每个网卡出厂时有一个全球唯一的 48 位地址：

```
00:1A:2B:3C:4D:5E
│           │
前 24 位    后 24 位（制造商自编）
(OUI, 厂商代码)
```

MAC 地址是**平的**——没有网络号/主机号之分，不能路由。它只在同一个局域网里有效。

### 以太网帧结构

```
┌───────┬──────────┬──────────┬────────┬──────┬──────┐
│前导码+│ 目的MAC   │  源MAC   │ 类型    │ 数据  │ FCS  │
│定界符  │  6 字节   │ 6 字节   │ 2 字节  │46~1500│ 4 字节│
│ 8 字节 │          │          │        │ 字节  │      │
└───────┴──────────┴──────────┴────────┴──────┴──────┘

类型字段：0x0800 = IPv4，0x0806 = ARP，0x86DD = IPv6
FCS：CRC-32 校验值
```

### 交换机 vs 集线器

**集线器（Hub）工作在物理层**——把收到的电信号原封不动复制到所有端口。所有端口共享同一个冲突域，带宽平分，A 和 B 同时发包就会碰撞。

**交换机（Switch）工作在数据链路层**——读取帧头里的 MAC 地址，查表做精确转发，只发给目标端口。每个端口是独立的冲突域，端口 1→2 和端口 3→4 可以同时通信，互不干扰。

交换机通过**自学习**建表：

```
1. 端口 1 收到源 MAC = AA:AA 的帧 → 记录"AA:AA 在端口 1"
2. 查表找目的 MAC = BB:BB → 找到在端口 2 → 只从端口 2 转发
3. 目的 MAC 不在表里 → 泛洪（Flooding）→ 正确的接收方回复 → 交换机学到它的位置
```

### ARP——IP 地址怎么找到对应的 MAC 地址？

你要给 `192.168.1.1` 发 IP 包，但以太网帧需要目的 MAC。你有 IP 但没有 MAC——ARP 就是解决"IP → MAC"翻译的。

```
1. 你广播 ARP 请求（目的 MAC = FF:FF:FF:FF:FF:FF）：
   "谁的 IP 是 192.168.1.1？告诉我你的 MAC"

2. 192.168.1.1 单播 ARP 响应：
   "我是 192.168.1.1，我的 MAC 是 BB:BB"

3. 你收到 → 记录到 ARP 缓存（arp -a 可查看）
   以后再用就不用问了
```

ARP 只在同一局域网内工作。目标 IP 不在本网段 → 对**默认网关的 IP** 发 ARP，拿到网关的 MAC，所有跨网流量都发给网关的 MAC，由网关做 IP 层转发。

---

## 总结

```
物理层：比特流。（没有边界，可能出错。）

链路层解决了五个连续的问题：

① 帧定界 → 比特填充，01111110 做边界标志
② 差错检测 → CRC，收方重算，对不上就丢弃
③ 可靠传输 → ARQ：停等 → GBN → SR
④ 介质访问控制 → CSMA/CD（有线）、CSMA/CA（无线）
⑤ 局域网寻址 → MAC 地址 + 交换机自学习 + ARP（IP→MAC 映射）
```

> 链路层在说："物理层给了我有噪声的比特流。我来把它组织成帧，检查错误，错了就重传，协调先后，最后准确送达局域网里的目标设备。但我解决不了跨局域网的问题——那是网络层的事。"
