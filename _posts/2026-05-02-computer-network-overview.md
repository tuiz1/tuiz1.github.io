## 起点：两台计算机想通信

计算机网络所有的复杂性，都来自一个简单的事实：**A 和 B 不在一起，但它们想交换信息。**

这个事实引爆了一系列必须解决的问题：

```
A 和 B 不在一起
  → 需要物理介质传信号（铜线、光纤、无线电波）
  → 介质有物理极限（距离衰减、热噪声、光速上限）
  → 需要差错检测（数据在传输中可能被噪声翻转）
  → 需要跨越多段链路的路由（互联网由无数子网组成）
  → 需要端到端的可靠性（丢包重传、拥塞控制）
  → 需要双方理解数据的含义（协议约定格式与语义）
```

计算机网络不是被天才凭空发明的，而是给定"A 和 B 想通信"这个前提，加上物理定律的约束，**一层一层必然推导出来的解决方案**。如果你重新设计一遍，大概率也会得到跟 TCP/IP 差不多的东西。

---

## 一、两个根本性的设计选择

在深入具体协议之前，有两个决定互联网命运的基本选择必须理解。

### 1.1 电路交换 vs 分组交换

19 世纪诞生的电话网采用的是**电路交换**——通话前先建立一条独占的物理通道，通话结束再释放。这条通道的资源全程归你一人所有，即使双方都在沉默，带宽照样空着。

计算机通信的流量模式与电话完全不同：

```
电话流量：持续恒定，一次通话几分钟
计算机流量：突发(Burst)，点击链接→加载→浏览→再点击

突发因子（峰值速率/平均速率）= 100:1 甚至 1000:1
```

如果用电路交换跑计算机流量：预留少了，高峰不够；预留多了，平时浪费。这个矛盾是不可调和的。

**分组交换**彻底解决了这个问题：数据被切成一个个"包 (Packet)"，每个包独立选择路径转发。链路的带宽被所有用户**动态共享**——你加载网页时给你 10Mbps，你阅读网页时的空闲带宽自动分配给别人。这就是**统计复用 (Statistical Multiplexing)** 的威力——不是按用户数量平分资源，而是按此时此刻的实际需求动态分配。

互联网选择了分组交换。代价是：网络层只做"尽力而为 (Best Effort)"的转发，不保证送达。可靠性交给端系统的 TCP 负责——这就是著名的**端到端原则 (End-to-End Argument)**。

### 1.2 端到端原则——智能终端 + 哑网络

传统电信的思路是让网络本身变得可靠（每一跳都确认、重传），但代价是网络设备变得极其复杂，协议僵化，难以演进。

互联网反其道而行之：**把复杂性放在端系统，让网络层保持简单。**

- 路由器只做一件事——根据 IP 地址查表转发，不关心数据内容，不保证可靠性
- 端系统（你的电脑和服务器）负责：丢包检测、重传请求、拥塞控制、数据加密

这个设计决策造就了互联网的惊人扩展性：核心路由器不需要理解 HTTP、DNS、TLS，它们只处理 IP 包。应用层协议可以独立演进（HTTP/1.1→HTTP/2→HTTP/3），网络层的路由器**零改动**。

## 二、协议——约定大于一切

两台计算机通信，光有物理连接不够。双方必须事先约定好三件事：

| 要素 | 回答的问题 | 举例 |
|---|---|---|
| **语法 (Syntax)** | "数据长什么样？字段怎么排列？" | 前 8 bit 是长度，接下来是数据 |
| **语义 (Semantics)** | "每个字段代表什么？收到某值要做什么？" | ACK=确认接收，NAK=请求重传 |
| **时序 (Timing)** | "谁先发？发完等多久？等不到怎么办？" | 发送方启动计时器，超时未收到 ACK 则重传 |

> 物理介质只能传输比特，但比特本身不携带含义。含义必须由发送方和接收方事先约定。这就是**协议为什么存在**的根本原因。

## 三、分层——计算机网络最核心的抽象

### 3.1 为什么不把一切交给一个程序做？

假设不做分层，让微信自己处理一切：

```
微信发一条消息给朋友，需要自己处理：
  ① 信号编码（网线上用多少伏特代表 1？）
  ② 帧定界（怎么知道一帧的开始和结束？）
  ③ 差错检测（收的比特对不对？）
  ④ 寻址（朋友 IP 是什么？怎么找到他？）
  ⑤ 路由选择（中间经过哪些网络？）
  ⑥ 可靠传输（丢了怎么办？乱序了怎么办？）
  ⑦ 加密（消息不能被人看到）
  ⑧ 会话管理（登录、退出、重连）
```

这是不可行的：每个应用都在重复造轮子；写微信的程序员需要懂物理层；换了传输介质（铜线→光纤），所有应用都要改代码。

分层的核心洞见：**不同层面的问题有完全不同的时间尺度和专业领域，必须由不同的人、在不同的抽象层次上分别解决。**

### 3.2 OSI 七层参考模型

国际标准化组织 (ISO) 将网络通信抽象为七层，每一层只在下面那层做不到的事情之上建立新能力：

```
┌─────────────────────────────────────────────┐
│ 7. 应用层   HTTP, DNS, SMTP, FTP            │  用户真正想要的服务
├─────────────────────────────────────────────┤
│ 6. 表示层   加密, 压缩, 编码转换             │  数据格式统一与翻译
├─────────────────────────────────────────────┤
│ 5. 会话层   对话控制, 同步点, 断点续传       │  管理通信会话
├─────────────────────────────────────────────┤
│ 4. 传输层   TCP, UDP                        │  端到端可靠/不可靠传输
├─────────────────────────────────────────────┤
│ 3. 网络层   IP, ICMP, 路由选择              │  寻址与跨网络路径选择
├─────────────────────────────────────────────┤
│ 2. 数据链路层 以太网, Wi-Fi, ARP            │  相邻节点间的可靠帧传输
├─────────────────────────────────────────────┤
│ 1. 物理层   网线, 光纤, 无线电               │  比特流与物理信号转换
└─────────────────────────────────────────────┘
```

**每一层存在的理由，一句话说到底**：

| 层 | 下面那层做不到的事 | 所以这层来做 |
|---|---|---|
| **数据链路层** | 物理层只认识连续的信号变化，不认识"帧" | 帧定界、差错检测 |
| **网络层** | 链路层 MAC 地址没有层级，无法全球路由 | 提供层次化地址 (IP)、跨网路由 |
| **传输层** | IP 只到主机、不到进程；IP 不可靠 | 提供端口号、可靠性 (TCP) |
| **应用层** | 传输层只传字节流，不知道字节含义 | 定义应用语义 (网页、邮件、文件) |

### 3.3 TCP/IP 四层模型——实际互联网的骨架

OSI 是先画好七层框框再让各组织填协议，TCP/IP 是先在 ARPANET 上跑通了再总结出模型。实际互联网只用四层：

```
OSI 七层              TCP/IP 四层
┌─────────┐          ┌─────────┐
│ 应用层   │          │         │
├─────────┤          │ 应用层   │  HTTP, DNS, SMTP
│ 表示层   │          │         │
├─────────┤          ├─────────┤
│ 会话层   │          │ 传输层   │  TCP, UDP
├─────────┤          ├─────────┤
│ 传输层   │          │ 网络层   │  IP, ICMP
├─────────┤          ├─────────┤
│ 网络层   │          │         │
├─────────┤          │ 网络接口层│  以太网, Wi-Fi
│ 数据链路层│         │         │
├─────────┤          └─────────┘
│ 物理层   │
└─────────┘
```

这是工程史上的经典教训：**可运行的代码战胜了完美的规范。** TCP/IP 的技术未必更优，但它简单、免费开源、先跑起来，最终覆盖了全球。

## 四、封装——分层在数据上的具体体现

每一层都在上层交给它的数据外面，**再套一层自己的信封（加自己的头部）**：

```
应用层：  [HTTP头 │ 数据]
传输层：  [TCP头  │ HTTP头 │ 数据]
网络层：  [IP头   │ TCP头  │ HTTP头 │ 数据]
链路层：  [ETH头  │ IP头   │ TCP头  │ HTTP头 │ 数据 │ ETH尾]
物理层：  把整帧转换成电/光/无线信号发出去
```

**封装的核心价值**在于中间设备的简洁性：

```
路由器收到一帧：
  1. 剥去以太网头 → 看到 IP 头
  2. 查路由表（只看 IP 头里的目的地址）
  3. 不改 IP 头，不改 TCP 头，不改 HTTP 头
  4. 加上新的以太网头（下一跳的 MAC 地址）
  5. 转发

路由器全程不需要知道里面是 TCP 还是 UDP，
更不需要知道是 HTTP 还是 FTP——它只看 IP 头。
```

这就是为什么中间设备可以极度简单、应用层协议可以独立演进。IPv4 升级到 IPv6，所有链路层交换机**零改动**——因为封装保证了每一层只看自己那一层的头。

## 五、物理层——信号、噪声与香农极限

物理层是整个协议栈里唯一真正触及"现实世界"的一层。它回答的是：怎么用物理信号表示 0 和 1？信号在路上衰减、被噪声污染后，每秒最多能无误传多少比特？接收方怎么知道每个比特从哪开始？

## 六、数据链路层——相邻节点的可靠传输

物理层给出的是无边界的、可能出错的比特流。数据链路层在此基础上建立"帧"的概念，解决帧定界、差错检测（CRC）、可靠传输（ARQ）、介质访问控制（CSMA/CD、CSMA/CA）和局域网寻址（MAC 地址、交换机自学习、ARP）五大问题。

## 七、网络层——全球寻址与路由（学习中）

链路层解决了局域网内的通信，但世界不只是一个局域网。网络层回答：怎么在几十亿台设备、几百万个网络中间，找到一台特定的设备？核心洞见是将 MAC 的"平坦地址"升级为 IP 的**层次化地址**——网络号 + 主机号，让路由聚合成为可能。目前正在学习分片、ICMP 和 NAT/IPv6。
